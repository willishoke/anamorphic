# anamorphic

Anamorphic is a tool for autonomous code generation through recursive problem decomposition. You give it a software problem in plain English — "build a URL shortener," "implement a rate limiter," "write a CSV processing pipeline" — and it uses Claude to break the problem down into a dependency tree of implementable subproblems, plans each one with a structured schema (functions, data structures, implementation steps, edge cases), and then generates working code for every leaf node in parallel. The result is a collection of files written to an output directory, with an optional git mode that gives each subproblem its own branch and streams periodic commits as the implementation is generated, so you can watch the code being written in real time.

The target user is a developer who wants to go from a rough problem statement to a working skeleton faster than they could scaffold it by hand, and who wants that skeleton to reflect a deliberate architectural decomposition rather than a flat dump of code. Anamorphic is not trying to replace a developer — the generated code is a starting point that expects a human to review, test, and iterate on. What it replaces is the tedious work of translating a mental model of a system into file structure, function signatures, and wiring. The interactive TUI lets you approve or push back on every decomposition and plan decision the LLM makes before any code is written, so you stay in control of the architecture.

The system is built as a React/Ink terminal application backed by a Node.js runtime. LLM calls use the Anthropic SDK when an `ANTHROPIC_API_KEY` is present and fall back to a local `claude` CLI subprocess otherwise. The traversal phase walks a BFS queue, asking Claude at each node whether the problem is small enough to implement directly (the threshold is roughly 120 lines, single responsibility) or whether it should be decomposed further. Once the full tree is built, a topological scheduler groups leaf nodes into epochs — batches of nodes whose dependencies have already been satisfied — and executes each epoch in parallel with a configurable worker limit. Git mode initializes a local repository in the output directory, creates a worktree per node, scaffolds a stub, and then streams the final implementation in chunks with periodic WIP commits before a final `feat:` commit closes the branch. A companion web dashboard at `localhost:7777` mirrors the TUI state over Server-Sent Events and is also deployable as a static demo to GitHub Pages.

What is complete and solid: the full decomposition loop with approval and refinement at every step; structured schema generation for leaf nodes; topological epoch scheduling; parallel code generation with and without git; the TUI across all five screens (input, root review, traversal, explore, build); and the web UI with real-time state sync. What is rougher or incomplete: the generated code targets Python and the 500-line cap is a soft prompt instruction rather than a hard constraint, so complex nodes can overrun it; dependency identification between sibling nodes is heuristic and LLM-driven, meaning it can miss or hallucinate edges; there is no retry or recovery path if a node fails during the build phase beyond logging the error; cycle detection in the scheduler is silent (cycles get bundled into a final epoch rather than surfaced to the user); and the tree lives only in memory, so there is no way to resume an interrupted session from where you left off beyond the explicit JSON export in the explore screen.
